## 当前结论
**当前结论：no-go（暂不建议进入正式预发 / 上线准备宣告）**

这次结论与上一轮不同的关键点不是“产品链路更差了”，而是**把错误测试方式从产品缺陷里剥离后，重新看清了真实门禁**：

- **已修正 / 不应继续误判的项**：旧版 smoke test 命中错误目标服务的问题，已不应再被算作“产品主链路坏掉”。
- **仍然存在的真实发布阻塞**：生产部署路径未唯一裁决、正式持久化方案未落定、OCR 生产依赖未验证、当前没有一份可复核的本轮 smoke / 回归通过证据。

因此，当前不应宣称“接近上线”或进入“只差收尾”的口径。

## 已验证事实

### 1. smoke test 方式已被纠正
已存在进程内 smoke test：`tests/smoke_test.py`

该脚本使用：
- `FastAPI TestClient`
- `from app.main import app`
- 进程内调用，不依赖外部端口

**判断**：
这说明“之前因为打错目标服务而失败”的结论应当作废，后续不能继续把该类问题算作产品主链路故障。

### 2. smoke test 已覆盖一条基础主链路
`tests/smoke_test.py` 当前覆盖以下路径：
- `/health`
- 注册 `/api/auth/register`
- 定价 `/api/pricing`
- providers `/api/providers`
- 简历上传 `/api/resume/upload`
- mock 支付 `/api/payment/create`
- 分析 `/api/analyze`
- 历史会话 `/api/sessions`
- 导出 DOCX / PDF `/api/export/{session_id}`

**判断**：
测试入口具备“主链路 smoke”雏形，门禁基础比上一轮更清晰。

### 3. 当前数据库实现仍为 SQLite
`app/db.py` 明确显示：
- `TESTING` 环境：`/tmp/test_ai_job_search.db`
- `VERCEL` 环境：`/tmp/data/ai_job_search.db`
- 其他环境：`data/ai_job_search.db`

**判断**：
当前生产态仍未脱离 SQLite；若走 Vercel，仍落在临时文件系统 `/tmp`。

### 4. 当前部署路径仍未收敛为唯一方案
仓库内同时存在：
- `vercel.json`
- `Dockerfile`
- `docker-compose.yml`（由既有风险文档可见）

其中：
- `vercel.json` 指向 `@vercel/python` + 静态构建
- `Dockerfile` 走容器化 `uvicorn`

**判断**：
当前依旧是双路径并存，不适合作为发布口径直接对外。

### 5. OCR 代码路径存在，但容器侧系统依赖未见安装
`requirements.txt` 包含：
- `pytesseract`
- `pypdfium2`
- `pdfplumber`

但 `Dockerfile` 仅执行：
- `pip install -r requirements.txt`

**未见**系统层 `tesseract` 安装。

**判断**：
OCR 能力不能直接视为已具备生产可用性。

### 6. 本轮未取得 smoke test 通过证据
本轮尝试执行：
- `python tests/smoke_test.py` → 本机无 `python` 命令
- `python3 tests/smoke_test.py` → 缺少 `fastapi` 依赖，无法运行

**判断**：
这不等于产品失败，但等于：**当前没有可复核的本轮通过证据**。发布把关不能用“理论上应当能过”替代“已实际跑通”。

## 已修复问题 vs 仍未解决风险

## 一、已修复 / 不应继续算作产品缺陷的项

### A. 旧 smoke test 打错目标服务
**问题**：之前失败中混入了错误目标服务问题。  
**现状**：已改为进程内自包含 `TestClient` 跑法。  
**结论**：后续不应再把这类失败直接算作“产品主链路损坏”。

### B. 当前项目已具备本地验证主链路的测试入口
**问题**：之前测试方式不稳定。  
**现状**：现在至少有一份明确的主链路 smoke 脚本。  
**结论**：测试方法论比上一轮更靠谱，风险评估应基于新脚本而不是旧结论。

---

## 二、仍未解决、且真实阻塞上线的风险

### P0-1 持久化方案未完成发布收口
**问题**：生产数据当前仍依赖 SQLite；Vercel 下写入 `/tmp/data`。  
**影响**：用户、支付、分析历史等关键数据不具备可靠持久化保证；实例切换、冷启动、多实例都会带来数据一致性风险。  
**结论**：这是**真实上线阻塞项**，不是测试误报。

### P0-2 唯一发布路径未裁决
**问题**：Vercel / Docker 两条路径并存。  
**影响**：测试结论无法稳定映射到生产环境；“本地通过”和“线上可发布”之间没有统一基线。  
**结论**：这是**真实发布门禁缺口**。

### P0-3 OCR 是否属于上线承诺能力尚未明确
**问题**：代码依赖存在，但运行环境依赖未验证；同时产品边界未收口。  
**影响**：扫描版 PDF / 图片简历可能出现“可上传但不可稳定解析”；若产品对外承诺 OCR，会形成体验违约。  
**结论**：若 OCR 属于首发能力，则当前为阻塞项；若不属于，必须在范围与界面提示上明确降级边界。

### P0-4 缺少本轮可复核的 smoke / 回归通过记录
**问题**：虽然 smoke test 脚本存在，但本轮没有成功执行并留档的通过结果。  
**影响**：Picard 无法基于“已验证证据”做 go 判断；只能基于代码静态事实做 conditional/no-go。  
**结论**：在发布门禁视角，这是阻塞项。

## 上线阻塞项

以下项在当前证据下应视为**上线阻塞**：

1. **未裁决唯一发布路径**
   - 问题：Vercel / Docker 双轨并存
   - 影响：无法形成单一发布门禁

2. **未落定正式持久化方案**
   - 问题：生产数据仍依赖 SQLite，且 Vercel 下落 `/tmp`
   - 影响：数据不可靠，不适合正式上线承诺

3. **缺少本轮 smoke / 回归通过留档**
   - 问题：本轮无可复核 PASS 证据
   - 影响：无法给出 go / conditional-go 的执行性判断

4. **OCR 上线边界未收口**
   - 问题：能力承诺与运行依赖未统一
   - 影响：易形成首发能力口径失真
