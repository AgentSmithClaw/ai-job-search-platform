# Spock P1 Spec（MVP 收口版）

更新时间：2026-03-26 16:50 CST  
负责人：Spock（产品/规格）  
主产物：`docs/handoff/spock-p1-spec.md`

## 1. 问题定义
GapPilot 当前已经具备“上传简历 + 输入 JD + 生成分析结果”的工程雏形，但离“可稳定交付”的产品状态还差一层范围收口：

- 当前功能点较多，但首发范围还没有被清晰约束
- 前后端已经能覆盖主链路，但产品口径仍可能发散
- 若继续并行推进全部功能，容易拖慢上线前收口

因此，本轮 P1 spec 的目标不是扩张功能，而是明确：
**首发版本到底交付什么、不交付什么、以什么标准算可验收。**

---

## 2. 当前 MVP 范围（本轮建议确认）

### 2.1 核心价值
帮助求职者完成一条最小但完整的闭环：
1. 上传/粘贴简历
2. 粘贴目标岗位 JD
3. 获得 AI 差距分析报告
4. 查看风险、优势、差距项与下一步建议
5. 可把目标岗位记录到投递管理里继续跟进

### 2.2 本轮纳入 MVP 的功能

#### A. 差距分析主流程（P0/P1）
- 用户注册/获取 access token
- 上传简历（至少支持 txt / docx / 文字型 pdf）
- 输入目标岗位与 JD
- 发起 gap analysis
- 返回结构化分析结果：
  - 匹配度分数
  - summary
  - strengths
  - risks
  - gaps
  - learning plan
  - interview focus
  - next actions
- 保存分析历史
- 支持导出报告（DOCX / PDF）

#### B. 投递管理最小闭环（P1）
- 新建投递记录
- 查看投递列表
- 更新投递状态
- 删除投递记录

#### C. 基础账户/额度能力（P0）
- 注册/登录式 access token 获取
- mock 支付购买 credits
- 分析消耗 credits
- 用户可看到基本 dashboard 数据

#### D. 最小 Dashboard（P1）
Dashboard 在本轮不是“完整运营工作台”，而是：
- 展示分析统计
- 展示最近分析记录
- 展示基础投递概览
- 能正确跳转到分析详情 / 核心页面

---

## 3. 当前明确非范围（本轮不要混入）
以下功能可以保留代码基础，但**不作为本轮首发完成定义**：

1. 支付宝 / 微信支付
2. 邮箱验证体系
3. 批量 JD 导入
4. 高级统计图表与经营看板
5. 完整设置中心
6. 复杂多模型编排配置 UI
7. OCR 的“生产级承诺”能力（除非 Picard 单独确认纳入首发）
8. 设计系统重构 / 长期品牌升级
9. 多前端并行维护

---

## 4. 差距分析核心流程

### 4.1 用户主流程
1. 用户注册/进入系统
2. 上传简历或粘贴简历文本
3. 填写目标岗位名称
4. 粘贴 JD
5. 系统发起分析
6. 返回结构化报告
7. 用户查看：
   - 匹配度
   - 优势
   - 风险
   - 差距项
   - 学习建议
   - 面试准备重点
8. 用户可导出报告
9. 用户可将岗位继续纳入投递跟踪

### 4.2 产品上必须成立的关键点
- 输入门槛低：不要求复杂配置
- 输出结构稳定：不是一段纯自由文本
- 报告可保存、可回看、可导出
- credits 扣费逻辑可解释
- 若 LLM 不可用，应有 fallback 或明确失败提示

---

## 5. 投递管理最小闭环
本轮投递管理不追求 ATS 级复杂度，只要求形成可用闭环：

### 必须有
- 公司名
- 目标岗位
- 状态（如 interested / applied / interview / offer / rejected）
- 可编辑
- 可删除
- 可列表查看

### 可以暂时弱化
- 自动提醒
- 高级筛选
- 统计图
- 多轮面试详细结构化管理

---

## 6. 验收标准

## A. 差距分析主流程验收
满足以下条件可视为通过：
1. 用户可以成功注册
2. 用户可以上传至少一种简历文件并成功解析
3. 用户可以发起分析并得到结构化结果
4. 匹配度分数合法（0-100）
5. 分析结果可进入历史记录
6. 报告可以导出 DOCX / PDF

## B. 投递管理验收
1. 可以新增投递
2. 可以查看投递列表
3. 可以更新状态
4. 可以删除记录

## C. 账户与额度验收
1. mock 支付可成功增加 credits
2. 分析会正确扣减 credits
3. credits 不出现负数或超扣

## D. Dashboard 验收
1. Dashboard 不再只是纯静态 mock 展示
2. 至少接入分析统计与最近记录
3. 详情跳转链路正确
4. 具备基本 loading / empty / error 处理能力

---

## 7. 风险与依赖

### 风险 1：首发范围仍被继续扩张
- 影响：收口再次失败
- 建议：Picard 明确“本轮首发只保核心闭环，不扩张体验层功能”

### 风险 2：发布路径与持久化未裁决
- 影响：产品 spec 虽清晰，但无法转化为上线判断
- 依赖：La Forge / Worf / Picard 裁决

### 风险 3：OCR 能力边界不清
- 影响：用户可能误以为扫描件简历被稳定支持
- 建议：若不纳入首发，界面与口径同步降级说明

### 风险 4：前端主线尚未正式确认
- 影响：Dashboard、History、Tasks 等能力回收容易继续分叉
- 依赖：Picard 确认 `frontend/` 为唯一主线

---

## 8. 建议下一接手角色
- **Picard**：裁决首发范围、OCR 是否纳入、前端唯一主线
- **Sulu**：按本 spec 回收 Dashboard 数据态与关键页面
- **La Forge**：按本 spec 确保分析、支付、持久化、LLM fallback 能支撑最小闭环
- **Worf**：据此建立最小发布门禁

---

## 9. 给 Picard 的直接结论
**建议确认本轮 MVP = 差距分析主流程 + 投递管理最小闭环 + 基础额度体系 + 最小 Dashboard。**

只要这四块闭合，本轮就算形成“可交付前夜状态”；其他功能全部延后，不再混入本轮完成定义。
